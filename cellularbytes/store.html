The Vegetable Store

In this tutorial, we explore Encapsulation through the lens of a production-grade retail system. You will learn how to build self-contained objects that protect their own data, preventing the common state-management errors found in procedural code.

I. Complete Tutorial Source Code

class Vegetable:
    def __init__(self, name, price_per_kilo, initial_stock):
        self.name = name
        self.price_per_kilo = price_per_kilo
        self.stock = initial_stock 

    def sell(self, quantity):
        if quantity > self.stock:
            raise ValueError(f"Insufficient stock for {self.name}. Available: {self.stock}kg")
        self.stock -= quantity
        return quantity * self.price_per_kilo

    def get_info(self):
        return f"{self.name:<12} | Price: â‚±{self.price_per_kilo}/kg | Stock: {self.stock}kg"

class VegetableStore:
    def __init__(self):
        self.inventory = {
            "asparagus": Vegetable("Asparagus", 180, 100),
            "broccoli": Vegetable("Broccoli", 230, 100),
            "cabbage": Vegetable("Cabbage", 140, 100),
            "garlic": Vegetable("Garlic", 190, 100),
            "onion": Vegetable("Onion", 180, 100),
            "tomatoes": Vegetable("Tomatoes", 160, 100),
            "lettuce": Vegetable("Lettuce", 180, 100),
            "cauliflower": Vegetable("Cauliflower", 220, 100),
            "potatoes": Vegetable("Potatoes", 180, 100),
            "carrots": Vegetable("Carrots", 150, 100)
        }

    def display_menu(self):
        print("\n" + "="*45 + "\nGREENBYTES INVENTORY STATUS\n" + "="*45)
        for veg in self.inventory.values():
            print(veg.get_info())

    def process_transaction(self, item_name, quantity):
        name_key = item_name.lower()
        if name_key not in self.inventory:
            raise ValueError(f"'{item_name}' not found in database.")
        return self.inventory[name_key].sell(quantity)

def main():
    store = VegetableStore()
    while True:
        store.display_menu()
        choice = input("\nBuy (or 'exit'): ").strip()
        if choice.lower() == 'exit': break
        try:
            qty = float(input("Quantity in kilos: "))
            print(f"Total: â‚±{store.process_transaction(choice, qty):,.2f}")
        except ValueError as e:
            print(f"Error: {e}")

if __name__ == "__main__": main()

II. Architectural Deep-Dive

1. The Blueprint (Class Vegetable)
State Management & Encapsulation

At its core, the Vegetable class demonstrates true encapsulation â€” not just data hiding, but active protection of object invariants. The critical invariant here is stock >= 0. By making self.stock only modifiable through the sell() method, we prevent any external code from corrupting the object's state (e.g., setting negative stock). This is a foundational principle of "safe state" in professional software engineering. In large-scale systems, such encapsulation becomes the first line of defense against race conditions, invalid transactions, and data corruption.

Logical Validation Gates & Fail-Fast

The guard clause if quantity > self.stock followed by raise ValueError is a deliberate application of the Fail-Fast principle. We detect and reject invalid operations as early as possible â€” inside the object that owns the data. Raising exceptions enforces business rules at the lowest possible level, making bugs easier to trace and preventing silent failures (e.g., negative inventory that propagates through the system). In production code, you might extend this with custom exceptions like InsufficientStockError to allow more granular error handling by calling code.

Path to Greater Robustness

For even stronger encapsulation, consider making attributes private by convention (_stock) and exposing read-only access via @property. This pattern is widely used in libraries and frameworks to give controlled read access while completely protecting mutation paths.

2. The Orchestrator (Class VegetableStore)
Advanced Composition over Inheritance

VegetableStore is built using composition â€” it owns and coordinates a collection of Vegetable objects rather than inheriting from them. This "has-a" relationship is far more flexible than "is-a" inheritance. The dictionary-based self.inventory provides O(1) average-case lookup time thanks to Python's hash table implementation â€” critical when scaling from 10 items to 10,000+ SKUs without performance degradation.

Separation of Concerns & Loose Coupling

This class follows the Single Responsibility Principle and Open-Closed Principle. It knows how to manage the collection and delegate behavior, but has zero knowledge of how a vegetable computes price or updates stock. This clean separation means you can completely rewrite pricing logic, add discounts, taxes, or promotions inside Vegetable without ever touching VegetableStore. In real-world systems (e.g., e-commerce platforms), this kind of decoupling dramatically reduces maintenance cost and risk when requirements change.

Repository-like Pattern

The inventory dictionary acts as an in-memory repository. In production, this pattern is often replaced with database access (SQLite, PostgreSQL, Redis), but the interface remains the same â€” demonstrating how well-designed composition allows seamless evolution from prototype to enterprise system.

3. Runtime & Safety (The Main Loop)
Graceful Error Handling & EAFP

The try...except ValueError block embodies Python's "Easier to Ask Forgiveness than Permission" (EAFP) philosophy. Rather than pre-checking every possible input error, we let operations run and catch exceptions when they occur. This produces cleaner, more readable code and handles a wide range of failure modes (invalid numbers, insufficient stock, missing items) with minimal boilerplate. In contrast to "Look Before You Leap" (LBYL) approaches, EAFP is often more performant and maintainable.

User-Centric Design

By catching errors early and presenting friendly messages instead of stack traces, the loop creates a forgiving, professional user experience â€” a hallmark of mature software. In web or mobile applications, this same pattern translates to proper HTTP error responses and user feedback instead of 500 Internal Server Errors.
